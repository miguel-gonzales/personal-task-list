
# Taskminator Project Specification Report
**Version:** 1.0 (MVP Phase)
**Role:** Systems Architect & Product Specification

## 1. Vision Statement
**Taskminator** is a high-performance, cross-platform productivity engine for developers who value keyboard-centric speed and extreme simplicity. Unlike heavy enterprise project managers, Taskminator provides a "zero-friction" workflow through a lightning-fast CLI and a synced web companion, ensuring focus stays on the code, not the tool.

---

## 2. Product Scope & MVP Definition
The goal is to bridge the gap between local terminal efficiency and multi-device cloud synchronization.

* **Primary Interface:** Command Line Interface (CLI) for rapid input/output.
* **Secondary Interface:** Minimalist Web UI for task review and visibility.
* **Sync Engine:** Centralized Node.js API with a shared database.
* **MVP Goal:** A "Walking Skeleton" that supports CRUD operations across the CLI and Web without complex authentication for initial validation.

---

## 3. Data Model
The schema is designed for an "Offline-First" and "Last Write Wins" synchronization strategy.

| Field Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | PK | Unique identifier (client-side generatable) |
| `title` | String | Required, 280 char | The task description |
| `status` | Enum | 'todo', 'done' | State engine for tracking task progress |
| `created_at` | Timestamp | Required | Automatic creation timestamp |
| `updated_at` | Timestamp | Required | Used for sync conflict resolution |
| `deleted_at` | Timestamp | Nullable | Support for soft-delete/undo functionality |

---

## 4. API Specification (OpenAPI Summary)
The API follows RESTful principles to ensure the CLI remains lightweight and the Web UI remains responsive.

* **GET /tasks**: Returns the list of active tasks.
* **POST /tasks**: Creates a new task (UUID can be provided by CLI).
* **PATCH /tasks/{id}**: Partial updates for status toggling or title edits.
* **DELETE /tasks/{id}**: Sets `deleted_at` timestamp.

## 3. Core User Stories & Acceptance Criteria

### A. Zero-Friction CLI Task Entry
**Story:** As a developer, I want to add tasks via the terminal so I don't break my flow.
* **Given** the CLI is configured to point to the Taskminator API.
* **When** I run the command `tm add "Review pull request"`.
* **Then** the CLI should generate a unique UUID v4 for the task locally.
* **And** it should send a `POST` request to `/tasks` with the generated ID and title.

### B. Index-Based Completion (The "Zero-Friction" Logic)
**Story:** As a user, I want to mark tasks as done using simple numbers instead of UUIDs.
* **Given** I have recently run `tm list`.
* **And** the task "Fix bug" is mapped to index `1` in the local session cache.
* **When** I run the command `tm done 1`.
* **Then** the CLI should resolve index `1` to the full UUID stored in the local session file.
* **And** it should send a `PATCH` request to `/tasks/{uuid}` with `{"status": "done"}`.
* **And** I should see "Task 1 marked as completed".

### C. Web Visibility
**Story:** As a user, I want to see my CLI updates reflected on the web dashboard.
* **Given** I open the Web UI.
* **Then** I should see a "Terminal" themed dashboard.
* **And** tasks updated via the CLI should show their current status automatically based on the updated_at timestamp.

## 4. Technical Implementation Details

### UUID Generation & Collision Avoidance
- **Mechanism:** The CLI  uses UUID v4 (random generation).
- **Collision safety:** Collisions are statistically impossible for the scope of this application, allowing the CLI to safely act as the "Source of Truth" for IDs. This enables seamless offline creation.

### CLI Session Mapping
To ensure the user never has to manually provide a UUID, the CLI must manage a **Local Session Map**:
1.  **Sync:** Every time `tm list` is called, the CLI saves a JSON file at `~/.taskminator/session.json`.
2.  **Mapping:** This file maps simple integers to the UUIDs returned by the API (e.g., `{"1": "550e8400-e29b..."}`).
3.  **Resolution:** Command handlers (like `done` or `rm`) read this local map before making API calls.

### Data Model Schema
* **id:** UUID (Generated by client or server).
* **title:** String (Required, max 280 characters).
* **status:** Enum (`todo`, `done`).
* **created_at / updated_at:** Timestamps for sync resolution.
* **deleted_at:** Timestamp for soft-delete functionality.
